Tests the pure business logic functions that decide winners and durations.
No database required — these are standalone math/logic functions.

--- normalizeMatchupDurationSeconds (7 cases) ---
This function clamps user-supplied durations to a valid range (60s min, 86400s max).

  1. zero becomes default
     Input: 0  ->  Expect: 86400 (24 hours)
     Why: A zero duration is treated as "not set", so we fall back to the 24-hour default.

  2. negative becomes default
     Input: -5  ->  Expect: 86400
     Why: Negative values are nonsensical; same fallback applies.

  3. below min becomes min
     Input: 30  ->  Expect: 60 (1 minute)
     Why: We don't allow matchups shorter than 60 seconds.

  4. at min is unchanged
     Input: 60  ->  Expect: 60
     Why: Boundary check — exactly at the minimum should pass through.

  5. above max becomes max
     Input: 90001  ->  Expect: 86400
     Why: Can't have a matchup longer than 24 hours.

  6. at max is unchanged
     Input: 86400  ->  Expect: 86400
     Why: Boundary check — exactly at the maximum should pass through.

  7. valid passthrough
     Input: 3600  ->  Expect: 3600
     Why: A normal 1-hour duration should come out exactly as entered.

FILE 2: api/middlewares/rate_limit_test.go  (3 tests)
=======================================================
Tests the login endpoint rate limiter.
Each IP gets a bucket of 100 requests, then gets throttled.

  1. AllowsInitialRequests  (pre-existing test)
     Sends 5 POST /login requests from the same IP.
     Expects all 5 return HTTP 200.
     Why: Normal usage should never be blocked.

  2. BlocksAfterBurst  (new)
     Drains the entire 100-request burst from one IP (all return 200).
     Then sends the 101st request from the same IP.
     Expects HTTP 429 (Too Many Requests).
     Why: This is the actual enforcement check — without it, changing the
     burst size from 100 to 10000 would go unnoticed.

  3. UsesXForwardedFor  (new)
     Sends 100 requests with X-Forwarded-For: 1.2.3.4 (different RemoteAddr each time).
     Then sends a 101st request with the same X-Forwarded-For header.
     Expects HTTP 429.
     Why: Behind a load balancer, RemoteAddr is the LB's IP, not the client's.
     The limiter must key on X-Forwarded-For so real clients can't bypass it
     by cycling through LB addresses.

FILE 4: frontend/src/__tests__/useCountdown.test.js  (7 tests)
=======================================================
Tests the useCountdown hook that drives the live timer on every matchup and bracket round.
Uses Jest fake timers so we can control time without actually waiting.

  1. returns null and isExpired=false when targetTime is null
     Call useCountdown(null) — expect remainingMs=null, isExpired=false, formatted="0:00:00".
     Why: No end time means "no timer" — should be inert, not expired.

  2. returns 0 and isExpired=true for a past targetTime
     Pass a timestamp 1 hour in the past.
     Expect remainingMs=0, isExpired=true.
     Why: Timer already ran out — should show as expired immediately.

  3. returns positive remainingMs for a future targetTime
     Pass a timestamp 1 hour from now.
     Expect remainingMs > 0, isExpired=false.
     Why: Timer is live — basic sanity check.

  4. formats remaining time correctly
     Set target to exactly 1 hour + 5 minutes + 3 seconds from now.
     Expect formatted="1:05:03".
     Why: The formatted string is what's displayed to users. Padding must be correct.

  5. counts down when the interval fires
     Start a 10-second timer, advance fake time by 1 second.
     Expect remainingMs is now less than it was.
     Why: The interval should be ticking — if it's broken, countdowns never move.

  6. transitions to isExpired after time elapses
     Start a 2-second timer, advance fake time by 3 seconds.
     Expect isExpired=true, remainingMs=0.
     Why: Expiry triggers the "time's up" UI state. Must flip at the right moment.

  7. clears interval when targetTime becomes null
     Start a 5-second timer, then rerender the hook with targetTime=null.
     Expect remainingMs=null and isExpired=false.
     Why: If a matchup is reset or the component unmounts, the interval must be
     cleaned up — otherwise you get memory leaks and stale state updates.
